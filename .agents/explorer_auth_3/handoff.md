# Security and Authentication Analysis Report

This report presents a read-only investigation of API endpoint security, authentication implementation, and validation logic within the JobHunt Pro backend application.

---

## 1. Observation

### A. Route Identification (`/api/v1/*`)
Across the backend service (defined in `backend/main.py` and `backend/billing.py`), the following routes are defined under `/api/v1/*`:

1. **`POST /api/v1/scrape`**

   - *Code*: `@app.post("/api/v1/scrape", dependencies=[Depends(verify_jwt)])`
2. **`POST /api/v1/generate-cover-letter`**
   - *File*: `backend/main.py` (Line 60)
   - *Code*: `@app.post("/api/v1/generate-cover-letter", dependencies=[Depends(verify_jwt)])`
3. **`POST /api/v1/ai/generate-cover-letter/stream`**
   - *File*: `backend/main.py` (Line 65)
   - *Code*: `@app.post("/api/v1/ai/generate-cover-letter/stream", dependencies=[Depends(verify_jwt)])`
4. **`POST /api/v1/accounts`**
   - *File*: `backend/main.py` (Line 75)
   - *Code*: `@app.post("/api/v1/accounts", dependencies=[Depends(verify_jwt)])`
5. **`POST /api/v1/checkout`**

### B. Current Authentication Implementation
Authentication is built around JSON Web Tokens (JWT) using `HS256` symmetric signing.
- *File*: `backend/auth.py`
- *Algorithm*: Uses `jwt.decode` with `JWT_SECRET_KEY` (default: `"super-secret-key-jobhunt-pro"`).
- *Dependency*: `verify_jwt` is a FastAPI dependency wrapper:
  ```python
  security = HTTPBearer(auto_error=False)
  async def verify_jwt(credentials: HTTPAuthorizationCredentials = Security(security)) -> dict:
      if not credentials:
          raise HTTPException(status_code=401, detail="Authorization header missing or invalid scheme")
      ...
  ```
- *Routing enforcement*: Most routes selectively invoke this dependency using FastAPI's route decorator `dependencies=[Depends(verify_jwt)]`.

1. **Mock Endpoints (`tests/e2e/conftest.py`)**:

   - `GET /api/v1/auth/verify` (verifies token, Lines 49-51)
   - `GET /api/v1/dashboard/layout-config` (Line 90)
   - `GET /api/v1/dashboard/metrics` (Line 104)
   - `POST /api/v1/scraper/start` (Line 119)
   - `GET /api/v1/scraper/status/{task_id}` (Line 131)
   - `GET /api/v1/cicd/status` (Line 147)
   - `POST /api/v1/cicd/deploy` (Line 168)

2. **Test Runs**:
   - `python -m pytest tests/e2e/test_r4_auth.py tests/test_security_hardening.py`
     - **Result**: `18 passed, 15 warnings`
   - `python -m pytest tests/test_backend_secured.py`
     - **Result**: `4 failed, 5 passed`
     - **Failure details**: Tests `test_cover_letter_streaming_response` and `test_cover_letter_tone_support` fail with `AssertionError` (expecting `"stream-start"` but getting real LLM content).
     - **Cause**: The test uses `monkeypatch.setattr("backend.ai_engine.generate_smart_cover_letter_stream", ...)` but `backend/main.py` imports this function directly (`from .ai_engine import generate_smart_cover_letter_stream`) before monkeypatch runs.

3. **Security Hardening Expectations (`tests/test_security_hardening.py`)**:
     Tests target `web/app_v2.py` and enforce:
     - Blocking unauthorized calls to `/api/system/*` and `/api/multi-tenant/*` with HTTP 403 unless a valid `X-Cron-Secret` or query `key` matching `CRON_SECRET` is supplied.
     - Web Application Firewall (WAF) blocking and blackholing of malicious path probes (`/.env`, `/wp-admin/index.php`) and exploit payloads (SQL injection, XSS, directory traversal).
     - Response headers presence: `X-Frame-Options: DENY`, `X-Content-Type-Options: nosniff`, `X-XSS-Protection: 1; mode=block`, `Referrer-Policy: strict-origin-when-cross-origin`, and `Content-Security-Policy`.
     - Request Host validation: return 400 Bad Request if Host header is unrecognized/malicious.

## 2. Logic Chain

1. **Observed Vulnerability (Authentication Bypass)**:
   In `backend/billing.py`, `@router.post("/api/v1/checkout")` does not declare `Depends(verify_jwt)` or any other auth dependencies.
   - *Result*: Anyone can send checkout requests containing arbitary `user_id` values, bypass auth, and trigger checkout URLs or simulate Stripe activity.

2. **Observed Discrepancy (Production vs Test Endpoints)**:
   The E2E tests in `test_r4_auth.py` successfully authenticate because they hit the mock routes defined in `tests/e2e/conftest.py`. However, in a production setup, `/api/v1/auth/token` and `/api/v1/auth/verify` do not exist in the backend.
   - *Result*: Production authentication assumes that tokens are generated by an external identity provider or service and shared via the same `JWT_SECRET_KEY`.

3. **Observed Discrepancy (Security Hardening Gaps)**:
   While `web/app_v2.py` features robust Host validation, WAF controls, IP blackholing, and strict HTTP headers (tested by `test_security_hardening.py`), `backend/main.py` is bare and does not implement any of these safety checks.
   - *Result*: The API backend is highly vulnerable to scanning, host header attacks, and SQL injection/XSS probes if exposed directly.

---

## 3. Caveats

- **Scope Limit**: We only analyzed local backend and test files. The production deployment environment (e.g., reverse proxies, API gateways like Nginx or AWS API Gateway) was not examined. If the gateway blocks malicious inputs or enforces auth at the border, the backend gaps might be mitigated.
- **Assumptions**: We assume the backend is intended to be a standalone microservice where auth should be enforced locally rather than relying purely on an upstream proxy.

---

## 4. Conclusion

- **Unprotected Route**: `/api/v1/checkout` lacks auth and must be secured.
- **Backend Hardening**: `backend/main.py` requires security middleware to enforce HTTP security headers, valid Host headers, and basic query parameter sanitization.
- **Incorrect Test Mocking**: `tests/test_backend_secured.py` and `tests/test_backend.py` have invalid monkeypatching configurations targeting `backend.ai_engine` instead of `backend.main` for the stream generator.

### Proposed Authentication Strategy

To enforce JWT Bearer authentication globally or selectively on all `/api/v1/*` routes while letting public routes (and health checks) pass through, we propose two options:

#### Option 1: Centralized FastAPI Middleware (Recommended for Rapid, Non-Breaking Hardening)
Add a middleware layer to intercept and authenticate all matching requests before reaching the routers.

*Implementation Sketch*:
```python
# To be added to backend/main.py
from fastapi import Request
from fastapi.responses import JSONResponse
from .auth import verify_jwt
from fastapi.security import HTTPAuthorizationCredentials

@app.middleware("http")
async def global_api_v1_auth_middleware(request: Request, call_next):
    path = request.url.path
    if path.startswith("/api/v1/"):
        # Define exact public endpoint exemptions
        public_endpoints = {
            "/api/v1/auth/token",
            "/api/v1/health",
        }
        if path not in public_endpoints:
            auth_header = request.headers.get("Authorization")
            if not auth_header:
                return JSONResponse(
                    status_code=401,
                    content={"detail": "Authorization header missing or invalid scheme"}
                )
            
            parts = auth_header.split()
            if len(parts) != 2 or parts[0].lower() != "bearer":
                return JSONResponse(
                    status_code=401,
                    content={"detail": "Authorization header missing or invalid scheme"}
                )
            
            try:
                creds = HTTPAuthorizationCredentials(scheme="Bearer", credentials=parts[1])
                payload = await verify_jwt(creds)
                request.state.user = payload  # Make payload available to request handlers
            except Exception as e:
                from fastapi import HTTPException
                if isinstance(e, HTTPException):
                    return JSONResponse(status_code=e.status_code, content={"detail": e.detail})
                return JSONResponse(status_code=401, content={"detail": "Invalid token"})
                
    return await call_next(request)
```

#### Option 2: Idiomatic Router Refactoring (Recommended for Clean Architecture)
Decouple endpoint definitions from their raw paths and group them into public and secure routers.

1. **Public Router**:
   ```python
   public_api_router = APIRouter(prefix="/api/v1")
   
   @public_api_router.post("/auth/token")
   async def login(...):
       ...
   ```
2. **Secure Router**:
   ```python
   secure_api_router = APIRouter(prefix="/api/v1", dependencies=[Depends(verify_jwt)])
   
   # Mount sub-routers (e.g. billing, scraping) under the secure router
   secure_api_router.include_router(billing_router, prefix="/billing")
   secure_api_router.include_router(scrape_router, prefix="/scrape")
   ```
   *Note*: This requires changing path definitions inside `backend/billing.py` from `@router.post("/api/v1/checkout")` to `@router.post("/checkout")`.
