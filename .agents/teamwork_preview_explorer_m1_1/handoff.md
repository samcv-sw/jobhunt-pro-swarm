# Handoff Report: Backend Core Audit

## 1. Observation
This audit investigated the FastAPI backend core files, routers, database configuration, and selected core utility files. The following observations were recorded:

* **Insecure Direct Object Reference (IDOR)**:
  * File: `backend/billing.py`, lines 19-20:
    ```python
    @router.post("/api/v1/checkout", dependencies=[Depends(verify_jwt), Depends(rate_limiter)])
    async def create_checkout_session(request: CheckoutRequest):
    ```
    The endpoint takes `CheckoutRequest` with a user-supplied `user_id` inside it (line 41: `client_reference_id=request.user_id`), but never validates that it matches the authenticated user inside `verify_jwt`.

* **Missing Webhook Authentication**:
  * File: `backend/routers/webhooks.py`, lines 18 and 52:
    The endpoints `@router.post("/api/v1/webhooks/brevo")` and `@router.post("/api/v1/webhooks/sendgrid")` process bounce/spam reports and run updates (`UPDATE users SET email_bounced = 1 WHERE email IN ...`) without checking request signatures or verifying the client source.

* **HTML Injection / XSS in Email Preview**:
  * File: `backend/routers/emails.py`, lines 35-46:
    ```python
    html = f"""<!DOCTYPE html>
    <html lang="en">
    <head><meta charset="UTF-8"><style>
    body{{font-family:Arial,sans-serif;max-width:600px;margin:auto;padding:20px}}
    </style></head>
    <body>
    <p>{body.replace(chr(10), "<br>")}</p>
    ...
    ```
    The `body` content generated by the AI or user input is directly interpolated into the HTML template without escaping.

* **PostgreSQL Engine Incompatibility in Raw SQL**:
  * File: `backend/routers/analytics.py`, lines 115-122:
    ```sql
    SELECT
        COUNT(DISTINCT referrer_id) as total_referrers,
        COUNT(*) as total_referrals,
        SUM(CASE WHEN converted = 1 THEN 1 ELSE 0 END) as conversions
    FROM referral_tracking
    ```
    Comparing a boolean column `converted` to integer `1` is invalid in PostgreSQL.
  * File: `backend/routers/scraping.py`, lines 121-127:
    ```sql
    SELECT platform, COUNT(*) as total,
           SUM(CASE WHEN status = 'success' THEN 1 ELSE 0 END) as successes,
           SUM(CASE WHEN status = 'failed' THEN 1 ELSE 0 END) as failures
    FROM scrape_log
    WHERE created_at >= datetime('now', '-7 days')
    ```
    `datetime('now', '-7 days')` is SQLite-specific syntax and fails in PostgreSQL.

* **Garbage Collection (GC) Settings**:
  * File: `backend/main.py`, line 6:
    `gc.set_threshold(50, 5, 5)`
    This sets GC threshold to extremely low values (default is `700, 10, 10`).

* **Duplicate Auth Dependency**:
  * File: `backend/routers/referral.py`, line 20-21:
    ```python
    @router.post("/api/v1/referral/track", dependencies=[Depends(verify_jwt)])
    async def track_referral(req: ReferralRequest, payload: dict = Depends(verify_jwt)) -> dict:
    ```
    `Depends(verify_jwt)` is registered twice.

* **Database Connection Pools Conflict**:
  * File: `backend/database.py`, lines 123-128:
    `pool_size=3`, `max_overflow=2`.
  * File: `core/pg_sqlite_shim.py`, lines 525-529:
    `min_conn=1`, `max_conn=3`.
  Combined, these connection pools exceed Neon's free tier limit of 10 concurrent connections.

* **Missing `_record_failure` Calls (Dead Lockout Code)**:
  * File: `backend/auth.py`, line 138 defines `_record_failure(ip)`, but grep search returned no calls to it across active backend code.

* **Hardcoded CV Details in Fallback Templates**:
  * File: `core/cover_letter.py`, lines 76-83: Contains hardcoded resume details from Sam Salameh's CV.

* **Placeholders**:
  * File: `core/captcha_solver.py`, line 204: `# This is a placeholder for the audio fallback approach.` (returns `None`).
  * File: `core/compliance.py`, line 447: `return True  # Placeholder for production verification`.

## 2. Logic Chain
1. **Security Vulnerabilities**:
   * Since `request.user_id` is trusted directly to link Stripe checkouts, any user can forge a request body with a target victim's ID to pay for them, or trigger checkout webhooks affecting another user account (IDOR).
   * Since Brevo/SendGrid/Telegram webhooks lack authentication tokens/signatures, an attacker can spoof requests to mark legitimate users' emails as bounced or invoke bot actions.
   * Direct interpolation of raw text into HTML preview strings without escaping results in HTML Injection/XSS when rendered in the client.

2. **Database Failures**:
   * In PostgreSQL, boolean comparisons using `= 1` or date arithmetic using SQLite's `datetime()` function cause SQL grammar/type exceptions. Since the async SQLAlchemy database sessions execute these raw queries directly, these endpoints will crash in production PostgreSQL environments.

3. **Performance Degradation**:
   * Run-time garbage collection with a threshold of `50` causes GC sweeps to trigger constantly during minor allocations, causing severe latency spikes.
   * Duplicate dependencies in routers execute JWT decoding and database active-user validation twice per request.
   * Parallel SQLAlchemy and custom DB shim connection pools will exhaust the 10-connection limit on Neon, leading to connection drops.

---

## 3. Caveats
* The Celery broker (Redis) was not audited for configuration security or authentication.
* The Next.js frontend code is outside the scope of this backend audit; hence, client-side mitigations for the XSS/HTML Injection vulnerabilities were not checked.

---

## 4. Conclusion
The FastAPI backend core is generally clean but suffers from:
1. Critical security bugs (Stripe IDOR, unauthenticated webhooks, and XSS).
2. Portability errors (SQLite-specific queries that will crash on Postgres).
3. Severe performance bottlenecks (ultra-low GC threshold, connection pool conflicts, and duplicate dependency executions).
4. SaaS Multi-tenancy issues (hardcoded resume details in fallback templates).

Implementing the proposed corrections in `analysis.md` will resolve these issues.

---

## 5. Verification Method
* **SQL Compatibility**:
  * Run pytest tests or inspect the raw queries in `backend/routers/analytics.py` and `backend/routers/scraping.py` against a Postgres test database.
* **IDOR Validation**:
  * Submit a POST request to `/api/v1/checkout` with a token for User A, but specify `user_id` of User B in the body. If the returned URL points to User B's checkout session, IDOR is present.
* **GC Threshold**:
  * Run `import gc; print(gc.get_threshold())` inside the running app container. If it returns `(50, 5, 5)`, it is degraded.
* **Webhook Signature Validation**:
  * Send a mock POST to `/api/v1/webhooks/brevo` or `/api/v1/webhooks/sendgrid` from an external client without a signature/token. If it returns `200 OK` and marks emails as bounced in the DB, the vulnerability is active.
