# Analysis: Stealth Scraper Hardening and Parsing Robustness

## 1. Executive Summary
This report analyzes `scrapers/stealth_ingest.py` and its corresponding test suite `tests/e2e/test_r3_scraper.py`. It identifies multiple limitations in the current anti-bot bypass mechanism (such as invalid browser profile versions, header-profile mismatches, lack of browser fallback, and unsafe proxy configuration) and provides concrete recommendations to upgrade the scraper's stealth profile to defeat advanced protections like Cloudflare, JA3/TLS, and HTTP/2 fingerprinting. Furthermore, it outlines a robust method to extract structured job data (utilizing Schema.org JSON-LD parsing with a CSS selector fallback) instead of extracting raw HTML.

---

## 2. Codebase Overview and Context
- **Scraper Path**: `scrapers/stealth_ingest.py`
  - Utilizes `curl_cffi` (compiled wrapper around `curl-impersonate`) to spoof browser TLS/JA3 and HTTP/2 connection fingerprints.
  - Implements basic residential proxy rotation and session pinning using a `session_id`.
  - Performs a basic session warmup (accessing the root domain or robots.txt) and adds simple jitter delays.
  - Parses job details via `bs4.BeautifulSoup` using generic tags.
- **Test Suite Path**: `tests/e2e/test_r3_scraper.py`
  - Validates API integration with endpoints (`/api/v1/scraper/start`, `/api/v1/scraper/status/{task_id}`, and `/api/v1/scrape`).
  - Verifies that multiple stealth profiles are configured (at least 3, checking for `"chrome131"` and `"safari18_0"`).
  - Verifies structured database insertion of scraped job listings.

## 3. Limitations of the Current Anti-Bot Bypass Mechanism

### A. Outdated & Unsupported Browser Profiles
- **Problem**: The profiles list `STEALTH_PROFILES` includes `chrome146`, `safari2601`, and `firefox147`.
- **Impact**: Under the hood, `curl_cffi` relies on a compiled set of specific browser hello profiles. Future or invalid version strings (like `chrome146` or `safari2601`) will throw a `ValueError` at runtime or fall back silently to default libcurl TLS signatures, which are instantly blocked by Cloudflare, Akamai, or Datadome.
- **Verbatim code check**:
  ```python
  # scrapers/stealth_ingest.py:44
  "id": "chrome146",
  "impersonate": "chrome146",
  # scrapers/stealth_ingest.py:76
  "id": "safari2601",
  "impersonate": "safari2601",
  ```

### B. Header-to-TLS/JA3 Signature Mismatch
- **Problem**: Headers are statically defined and hardcoded inside `STEALTH_PROFILES`.
- **Impact**: If the hardcoded headers (like `User-Agent`, `Sec-CH-UA`, etc.) mismatch the TLS JA3/JA4 fingerprint or HTTP/2 frame properties generated by `curl_cffi` for the impersonated browser, WAF profiling engines will detect the discrepancy and block the request. E.g., Safari profiles sending `Upgrade-Insecure-Requests` or `Connection` headers under HTTP/2 triggers detection.

### C. Missing JavaScript/Captcha Execution Engine Fallback
- **Problem**: `curl_cffi` is a raw HTTP client and cannot execute JS. If a site presents a Cloudflare Turnstile, Datadome, or PerimeterX interactive challenge, the fetch will fail.
- **Impact**: The script has no fallback to browser automation. The project actually contains stealthy browser automation fallbacks in `core/stealth.py` (`NodriverFallback` and `ApexCamoufoxFallback`), but `stealth_ingest.py` fails to utilize them.

### D. Naive Cookie Management & Single-Use Session Overhead
- **Problem**: The current implementation initializes a new `AsyncSession` for every single URL fetched in `process_single_job`, generating new Google Analytics cookies (`_ga`, `_gid`) each time.
- **Impact**: Visiting a series of job pages on the same domain with completely fresh cookie sets and browser environments is a major bot indicator. Reusing a session and its cookie jar across multiple URLs is much more natural.

### E. Inefficient Warmup & Bot Fingerprint Triggers
- **Problem**: It fetches the home page or `/robots.txt` before *every single* target job URL.
- **Impact**: Normal users do not query `/robots.txt` right before opening a job page. Furthermore, repeating the warmup step for every page view creates double the bandwidth overhead and a suspicious 1:1 ratio of home-to-job requests.

### F. Unsafe X-Forwarded-For Spoofing
- **Problem**: If no proxies are configured, it injects a randomized client `X-Forwarded-For` header.
- **Impact**: Sending client-side `X-Forwarded-For` from non-proxy IP addresses is a well-known bot signal that modern WAFs flag immediately.

## 4. Hardening Recommendations for Advanced Anti-Bot Bypass

1. **Keep Impersonate Profiles Valid**: Restrict impersonate versions to those officially supported by the local `curl_cffi` version (e.g. `chrome120`, `chrome124`, `firefox120`, `safari18_0`).
2. **Dynamic Aligned Header Generation**: Remove hardcoded static headers. Instead, write a helper function (`_get_headers_for_profile`) to generate aligned headers matching the user-agent and TLS signature of the browser profile. Remove client hints and connection headers for Safari profiles.
3. **Session & Cookie Preservation**: Maintain a persistent `AsyncSession` per domain when scraping multiple URLs. Reuse cookies (like `cf_clearance`, Google Analytics cookies, and session IDs) to simulate a real return visitor.
4. **WAF Block Detection & Browser Fallback**: Implement string checks (e.g. searching for `"just a moment"`, `"turnstile"`, `"datadome"`, etc.) in the response. If blocked, automatically fall back to the browser automation drivers (`camoufox` or `nodriver` from `core/stealth.py`).
5. **Anti-Fingerprinting in Fallback Browser**: Ensure that fallback browsers spoof Canvas and WebGL contexts (using subpixel noise) and simulate human-like Bezier mouse trajectories (`HumanMouse`) to evade behavioral analysis.
6. **Decoy Traversal & Sequential Jitter**: Process URLs sequentially by domain, randomizing the crawling path, and implementing wider random delays (e.g. 5-15 seconds) to emulate human reading times.

## 5. Improving Parser to Return Robust Structured Data

The parser can be significantly hardened to return structured parsed data instead of relying on fragile tag-guessing:
1. **Schema.org JSON-LD Extraction**: Modern job portals wrap listings in `<script type="application/ld+json">` representing a `JobPosting`. The scraper should first search for and parse this tag. It yields 100% accurate, standardized fields (`title`, `hiringOrganization`, `description`, etc.).
2. **Refined CSS Selectors Registry**: If JSON-LD is missing, fall back to BeautifulSoup using a prioritized list of specific selectors for popular job sites (e.g. `.jobsearch-JobInfoHeader-title`, `h1.job-title`).
3. **Normalize & Clean Output**: Clean the extracted title (strip portal suffixes like `" - Indeed"`), strip HTML tags from descriptions, and validate that the output contains at least a valid non-empty `title` and `url` before returning.

```python
STEALTH_PROFILES = [

]

def _get_headers_for_profile(profile: dict, root_domain: str) -> dict:
    parsed = urllib.parse.urlparse(root_domain)
    host = parsed.netloc
    
    headers = {
        "Accept": "text/html,application/xhtml+xml,application/xml;q=0.9,image/avif,image/webp,*/*;q=0.8",
        "Accept-Language": "en-US,en;q=0.9",
        "Upgrade-Insecure-Requests": "1",
        "User-Agent": profile["user_agent"],
        "Host": host,
    }
    
    impersonate_str = profile["impersonate"]
    if "chrome" in impersonate_str:
        headers["Sec-CH-UA"] = f'"Google Chrome";v="124", "Chromium";v="124", "Not_A Brand";v="24"'
        headers["Sec-CH-UA-Mobile"] = "?0"
        headers["Sec-CH-UA-Platform"] = '"Windows"'
        headers["Connection"] = "keep-alive"
    elif "safari" in impersonate_str:
        headers.pop("Connection", None) # Safari HTTP/2 compatibility
        
    return headers
```
